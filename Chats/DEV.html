Joe·31/12/2025 1:45:59 PM
#1: SKU Numbering Strategies

There are 2 basic schools of thought on part (sku) numbering; one where the first few digits are a product ID and all the variants are just a serialized concatenation with the ID. The second is where the number is metadata - data about data encoded into the data itself; each digit contains information to specifically identify the variant.

Both are equally prevalent however personally and professionally, I lean strongly to the latter.

A great example of this is VIN code - vehicle identification numbers which also adds item serialization into the code as a suffix.

goodcar.com — How to Decode a VIN Number
Joe·31/12/2025 1:46:45 PM
#2: Inventory Types - BoM vs Formula-Based

Beyond the SKU, there are also a couple of inventory types; Bill of materials and formula based.

A BoM can be a discrete item or an assembly of discrete or sub-assemblies. The assembly can be broken down into atomic level inventory item.

In formula based inventory (think cake mix), once the discrete items are combined there is no option to disassemble the combined item. In this case, there are strong arguments to retain batch information on the discrete components of the formula.

Case in point: some years ago Tylenol had to recall 100% of the product because they found contamination in one batch of a competent chemical and could not trace which final products contained chemicals from this batch. There are hundreds of nutraceutical products sold at health food retailers that lack this capability as inventory software with this capability is both rare and expensive.
Joe·31/12/2025 1:47:19 PM
#3: Item Serialization

Item serialization is another aspect that comes to play in many inventory scenarios especially on bigger ticket items and assemblies
Joe·31/12/2025 1:47:38 PM
#4: RMA (Return Material Authorization)

And in many inventory systems out there, they neglected to account for RMA Return Mateial Authorization - to put things back in stock. Sometimes laws do not allow a returned product, even unopened, to be sold as new (US commerce law on computers for instance) so an inventory designation needs to exist to account for this possibility

Kirk·31/12/2025 10:31:35 PM
Basic rules for team development in a DEV environment

  1. Duplicate the script or layout before editing. This is your validated backup / recovery point. Optionally, add a BACKUP suffix to the name
  2. Edit in the original; as the internal ID is the basis, then the changes will be linked to everything else. Optionally, add WIP (Work-In-Process) suffix to the name
  3. If the changes are validated, you can optionally delete the BACKUP
  4. If the changes have NOT been validated and a code freeze needs to occur, duplicate the edited copy, then copy the CONTENT of the script or layout from the backup, onto the original, and you'll be back to the starting point.

These efforts are a safety net for EXISTING CODE, to keep fixes/updates that may not be workable (yet) from screwing up what was working, or at least giving you a restore point, and still work within the existing workflow.

For PRODUCTION LIVE CODE edits, everything is the same EXCEPT that you edit in the copy (so as to not disrupt production), and manually deal with connections to other places in testing the result. Once working, you can copy the CONTENT of the copy into the original script or layout.

Another partial workable option: Make a DDR prior to every change session and, using FMPerception, the code (not the layouts) from the DDR can be copied back into the solution.

🔄 Extract — DEV → TEST → PRODUCTION Workflow
Kirk·Fri Jan 2 2026 5:05 PM
Typically in the traditional Dev→Test/Production server structure (supported by Claris's 3 server license with any multi-user setup),

  1. DEV is where developers make changes
  2. when a freeze is ready, that indicated that code review and some preliminary - typically UNIT but sometimes SYSTEM test is complete
  3. A BUILD is pushed to TEST often with recent production data orignally seeded on TEST
  4. Test is performed
  5. if all defined validation is successful, the DEV BUILD version tested, is pushed to Production.
Kirk·Fri Jan 2 2026 5:06 PM
Note that OTTODeploy contains features to allow a quick code rollback to the prior version.
Brief Summary
Professional Release Workflow
Kirk is describing the standard way teams safely move FileMaker changes into the live system, using three environments: DEV, TEST, and PRODUCTION. The goal is to reduce risk, prevent broken updates from reaching users, and make sure changes are verified before they go live.

How the flow works (the main idea):
  • DEV (Development): Developers build and change things here.
  • Freeze moment: Work is considered ready for review and early testing.
  • Code review + early testing: Unit testing, and sometimes system testing.
  • Build goes to TEST: A specific build moves from DEV into TEST, often with seeded production-like data.
  • Testing happens in TEST: Validate features and acceptance checks.
  • Push to PRODUCTION: The same tested build is deployed to the live system.
Extra point (safety): OttoDeploy can do quick rollback to the prior version if a deployment causes issues.

Purpose: Controlled releases that are tested, repeatable, and reversible. This prevents the "it worked on my machine" problem and ensures production stability.
