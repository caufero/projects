SQL 101
Joe·4:11 PM
WHAT IS IT?

Structured Query Language (SQL) is a standardized programming language used to manage and manipulate data in relational database management systems (RDBMS). It allows users to perform operations such as querying, updating, and managing data stored in tables.

SQL was invented in the 1970s based on the relational data model. It was initially known as the structured English query language (SEQUEL). The term was later shortened to SQL. Oracle, formerly known as Relational Software, became the first vendor to offer a commercial SQL relational database management system.
Joe·4:13 PM
DO I NEED to KNOW SQL to USE FILEMAKER

No. The database engine that is built into FileMaker is called Draco and it is NOT a SQL compliant database.
Joe·4:18 PM
WHAT are SOME COMMON SQL DATABASES?

MySQL, PostgresSQL, Microsoft SQL Server, Oracle Database, SQLite and MS Access are the most common ones you will hear about.
Brief Summary
SQL 101 - Building Shared Vocabulary
Joe gives the team a quick "SQL 101" so everyone speaks the same language when database topics come up.
  • What SQL is: SQL means Structured Query Language. It's the common language used to talk to relational databases (databases that store data in tables, like rows and columns). With SQL you can read data (queries), add data, edit data, and delete data in tables. Joe adds history: SQL became popular from the 1970s, starting with an earlier name "SEQUEL."
  • Do you need SQL to use FileMaker? Joe's point: No, you don't need SQL to be productive in FileMaker, because FileMaker is not built like MySQL or PostgreSQL where SQL is the main way you work. You can build FileMaker apps using tables, relationships, layouts, scripts, and calculations without needing to be an SQL person.
  • Common SQL databases: He lists popular SQL systems (MySQL, PostgreSQL, SQL Server, Oracle, SQLite, Access) so when someone mentions them later, everybody knows those are the classic SQL database platforms.
  • Purpose: Joe is building shared vocabulary: what SQL is, how it fits in the database world, and where FileMaker sits compared to traditional SQL databases.
🔧 Friday-Saturday, January 2-3, 2026
SQL in FileMaker
Joe·10:34 PM
WHY are you DISCUSSING SQL in a FILEMAKER FORUM?

  • Because SQL can be very powerful in FileMaker.
  • Claris added SQL query abilities to FileMaker years ago.
Joe·10:47 PM
WHY add SQL if FileMaker's Draco engine is not SQL-compliant?

Joe gives 3 reasons:

  1. Familiarity: Many devs already know SQL, so it helps new devs join FileMaker.
  2. Compatibility: FileMaker can connect to external SQL data sources via ODBC/JDBC, so SQL matters there.
  3. Ease of Use (Context Relief): FileMaker normally requires layout context and relationships to access data. SQL can sometimes bypass "context problems" for small tasks and save time.
Joe·10:50 PM
HOW do I use SQL in FileMaker?

Two options:
  • ExecuteSQL (function)
  • Execute SQL (script step)
Joe·11:01 PM
Do they do the same thing?

ExecuteSQL (function)
  • Works on FileMaker tables
  • Can only READ data (SELECT)
  • Returns results as text into a variable or calculation
  • Cannot INSERT, UPDATE, or DELETE FileMaker records
  • Fast when used correctly and avoids layout context
Execute SQL (script step)
  • Works on external SQL sources only (ODBC/JDBC data sources)
  • Can do full CRUD (Create/Read/Update/Delete)
  • Does not work on FileMaker native tables
Joe·11:54 PM
Why do some long-time FileMaker devs speak negatively about SQL in FileMaker?

Joe's main point:
  • Many older FileMaker devs never learned SQL because for decades FileMaker worked fine without it.
  • ExecuteSQL for FileMaker tables only arrived in FileMaker 12 (2012), so lots of people built habits long before it existed.
  • Some negativity is "fear of learning or comfort with old ways," not always real technical reasons.

Joe gives historical context:
  • FileMaker was non-relational for a long time.
  • Relationship graph arrived later (he mentions v6 era).
  • ExecuteSQL (for FileMaker tables) came in v12.

Joe mentions key community resources:
  • "Missing FM12 ExecuteSQL Reference" (Beverly Voth and Kevin Frank) on FileMakerHacks.
  • Wim Decorte's work and presentations.
Joe·Late night
Known ExecuteSQL performance issue (what Wim highlighted)

  • If the current user is editing a record in a table and it is not committed yet, and you run ExecuteSQL against that same table, FileMaker may have to pull a lot more data to the client to run the query.
  • That can make a query extremely slow on very large tables (example: hundreds of thousands of records).
  • Many people heard "ExecuteSQL can be slow" without understanding the specific situation, and that created fear.

Joe's advice:
  • Don't accept forum opinions blindly.
  • Do your own research and learn the tool properly.
Brief Summary
SQL Tools in FileMaker
  1. SQL is useful in FileMaker, even though FileMaker's own database engine is not a full SQL database. Claris added SQL features because many developers already know SQL, because FileMaker can connect to external SQL databases, and because SQL can help you avoid FileMaker context issues in some cases.
  2. There are two different SQL features in FileMaker:
    • ExecuteSQL (function): works on FileMaker tables, but it can only READ data (SELECT). No insert, no update, no delete.
    • Execute SQL (script step): can do full CRUD, but only for external SQL sources through ODBC/JDBC, not FileMaker tables.
  3. Some people talk badly about ExecuteSQL because:
    • Many older FileMaker devs never had SQL inside FileMaker for many years, so they avoided it.
    • There is a real performance trap, but it is specific: if the current user has an uncommitted record open and you run ExecuteSQL on that same table, ExecuteSQL can become extremely slow on big tables.

USING SQL in FILEMAKER - Understanding the DRACO Engine

Draco is not a traditional SQL database engine. Instead, it is built on Apache NiFi, which is a dataflow system designed for processing and distributing data. While it supports SQL-like operations, it does not fully comply with SQL standards.

SQL Support in Draco
  • SQL-92 Implementation: Draco supports a SQL-92 implementation for query translation, allowing some SQL-based interactions.
  • CRUD Operations: Full Create, Read, Update, and Delete (CRUD) capabilities are typically accessed through plug-ins rather than natively.
Limitations
  • Not a SQL Database: Draco is primarily a data processing system, not a SQL-compliant database engine.
  • Dependency on Plug-ins: SQL functionalities are often reliant on external plug-ins, which may not cover all SQL features.

In summary, while Draco can handle SQL-like queries, it is not a fully SQL-compliant database engine.

Here is a presentation specifically on the Draco database engine from the 2016 DEVCON by the then Chief Software Architect at FileMaker, ie Clay Maeckel.

community.claris.com — Draco presentation file

I cannot say that I personally understand all of the technical ideas covered in this presentation but I understand enough to be able to say that the DRACO engine that drives FileMaker is an awesome database engine. What it is able to do for us across the board is truly a work of art in engineering. So please understand that while it is true that Draco is not fully SQL compliant, this does not mean it is an inferior database. The non-SQL compliance does not affect its internal performance. And the internal performance is quite impressive for all the various input it can and does handle.

Because of the power of the Draco engine and its fine tuning, we do not need ExecuteSQL to make the database engine run faster. It's fast and works really well with it's built in reads, searches and filters. In fact, and I can't be sure of this but there's a possibility, that even when actually use the ExecuteSQL function to extract data, it's possible that under the hood the script takes the parameters and translates or transforms our query request into search or find requests that are native to Draco. Don't quote me on that but it's possible. I just don't know enough of how FM interprets and uses scripts under the hood.

Brief Summary
Team Argument: Trade-offs in Tool Choice
This is a team argument about whether it is wise to use ExecuteSQL inside FileMaker. Nobody is arguing that FileMaker "cannot" do something without SQL. The fight is about trade-offs: speed and convenience versus simplicity and long-term maintainability.
  • Kirk's main point: In his experience, FileMaker's native tools already cover the real jobs, so he has never "needed" SQL. Bringing SQL into FileMaker can make a solution feel like mixing two languages in one document. If everyone on the team is not fluent in both, the code becomes harder to read, harder to maintain, and more expensive over time. Only add another language when the benefit is clear: minimum diversity = maximum manageability = lowest cost.
  • Joe's main point: The goal is using FileMaker to its fullest. He treats ExecuteSQL like a normal FileMaker function that happens to use SQL syntax, and it can be a neat shortcut. Like a specialty tool: most jobs use standard tools, then one job needs a special tool.

💡 Extract — HOW, WHEN and WHY to USE EXECUTE SQL
Joe·Jan 3, 2026
HOW, WHEN and WHY to USE EXECUTE SQL

The short answer to this is that if you understand what I've written above, which I know is a lot, but if you understand the essence of it, then I don't have to really tell you when to use ExecuteSQL because the short answer is: use it anytime you want to.

But see that's the key: knowledge. And that's true of any tool. There's an old saying that goes: "When you first learn how to use a hammer, everything out there looks like a nail."

Nowhere is this more true than with ExecuteSQL. When I first realized that I could avoid context and get data without having to switch layouts, I used ExecuteSQL everywhere. It was my hammer and I paid a price for it. But if you know anybody who hasn't make any mistakes in learning a new programming language or syntax let me know. And in fact it was this painful experience that led me to learn more about ExecuteSQL and this has paid dividends.

Now I cannot list all the best ways or times to use ExecuteSQL because I dont them all but I will put forth a couple of good uses for ExecuteSQL. Also maybe Cyril and Oz to could add a few situations where they have found it useful and that should round off this brief introduction to using SQL in FileMaker.

Let me start with the conclusion to the article from Wim Decorte of Soliant Consulting that I referenced above:
Wim Decorte conclusion:

"FileMaker ExecuteSQL() is an incredibly useful tool in our tool belt, and I would strongly encourage FileMaker developers to use it where it fits. It does some very unique things extremely well provided you stay within its performance envelope. It is not a very wide envelope, but do not let that discourage you or shunt it.

The Good is really good; the Bad and the Ugly can mostly be avoided."
Joe·Continued
Good use cases:

  1. To get one piece of information from another table using a variable based on a field in your current context without needing to change the Graph or open a Window offscreen, perform a find and then close the offscreen window. It's quick, easy and fast.
  2. To retrieve an ordered list of any subset of data without needing to add a table to your graph that might need a global constant and or a global match field to get to that list of records. Useful in a script step where you will consume that list in order to display something to a user.
  3. To retrieve an aggregate of a value (like Count or Sum) without needing to have a context like you would with using the Count or Sum functions in a script step. Useful for building summary fields on the fly without needing context or a graph relationship. With a known context, Aggregate functions in FileMaker are just as efficient.
  4. To build a virtual list on the fly. Useful when you need to get data from more than one unrelated table. You can avoid opening and closing of windows and other mechanics.
  5. To quickly build very specialized and complicated dashboards. Example: from 6 hours to 6 minutes with ExecuteSQL

He adds a caution: it can create issues like record locking or performance trouble if used carelessly, so use it deliberately.
Brief Summary
Practical Guidance & Use Cases
Joe switches from debate mode to practical teaching. He says ExecuteSQL is safe to use anytime you want only if you actually understand it, because new tools can tempt you to overuse them and cause problems. He admits he once used ExecuteSQL everywhere just to avoid context switching, and that mistake forced him to learn its limits.
  • Five main situations: grabbing a single value without context switching, getting an ordered subset list without extra relationships or globals, calculating quick aggregates without needing summary context, building virtual lists without loops or complex graph setups, and powering specialised dashboards or WebViewer data feeds where the right query can dramatically reduce runtime.
  • Warning: It can create issues like record locking or performance trouble if used carelessly, and he points to Wim Decorte's guidance: ExecuteSQL is extremely useful inside its performance limits, and most bad or ugly outcomes can be avoided when you know what you're doing.
