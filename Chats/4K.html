Kirk·1:34 PM
There are 2 basic schools of thought on part (SKU) numbering: one where the first few digits are a product ID and all the variants are just a serialized concatenation. The second is where the number is metadata — each digit contains information to specifically identify the variant.

Both are equally prevalent, however personally and professionally I lean strongly to the latter.

A great example of this is VIN code (vehicle identification numbers) which also adds item serialization into the code as a suffix.
goodcar.com — How to Decode a VIN Number
Kirk·1:45 PM
Beyond the SKU, there are also a couple of inventory types: Bill of Materials and formula-based.

A BoM can be a discrete item or an assembly of discrete or sub-assemblies. In formula-based inventory (think cake mix), once the discrete items are combined there is no option to disassemble the combined item. In this case, there are strong arguments to retain batch information on the discrete components.

Case in point: Tylenol had to recall 100% of the product because they found contamination in one batch of a component chemical and could not trace which final products contained chemicals from that batch.
Kirk·1:47 PM
Item serialization is another aspect that comes to play in many inventory scenarios especially on bigger ticket items and assemblies.
Kirk·1:51 PM
And in many inventory systems out there, they neglected to account for RMA (Return Material Authorization) to put things back in stock. Sometimes laws do not allow a returned product, even unopened, to be sold as new — so an inventory designation needs to exist to account for this possibility.

🔗 Extract — OData: Features, Why It's Useful & How to Learn It for FileMaker
Joe·30 Dec 2025 · 10:22 PM
Key Features of OData
  • Standardized Query Language: OData uses a query language that enables clients to request specific pieces of information from a server.
  • RESTful Protocol: Based on REST principles — uses standard HTTP verbs: GET, POST, PUT, and DELETE.
  • Data Access: Allows clients to access data from various sources like databases or cloud services using a uniform URL structure.
  • Metadata: OData provides metadata that describes the data model, making it easier for developers to understand how to query and interact with the data.
Joe·30 Dec 2025 · 10:24 PM
How OData is Used
  • Data Retrieval: Applications use OData to request specific data from a server, such as filtered data sets or just a certain number of records.
  • Data Manipulation: Allows not only for reading data but also for creating, updating, and deleting entries on the server.
  • Integration: OData is often used to integrate disparate systems, allowing them to share data without needing custom integration code for each system.
  • Compatibility: Many enterprise systems and tools, like Microsoft Excel and Power BI, support OData natively.
Joe·30 Dec 2025 · 10:47 PM
Q: OData seems very similar to an API. What makes it different and useful to us?

A: OData endpoints are more prevalent than typical API endpoints and much easier to code for developers on both ends of the transaction. One of the biggest advantages is that you can specify more precisely exactly the data you are looking for.

Example: Say you are querying a database with 15,000 undergraduates and looking for all females with a GPA of 3.2 or better. A standard API would return the whole record for each of those ~3,000 students — say 40 fields each — giving you 120,000 pieces of unsorted information that you then have to parse. Using OData you can specify in the GET call that you only want 3 fields (ID, Name, GPA) sorted descending by GPA. The data comes back already transformed and ready to use.
Joe·30 Dec 2025 · 11:24 PM

Brief Summary
OData Explained for the Team
Joe explains what OData is, why it matters, and how to learn it for FileMaker.
  • What it is: A standard way to access and work with data over the web. Uses normal HTTP requests (GET, POST, PUT, DELETE) like a typical API, plus a consistent query system and metadata.
  • Why it's better than a plain API: You can request only the exact fields you want, with filtering and sorting built into the same call — meaning smaller JSON payloads and less cleanup work on the client side.
  • How to learn it: Joe points to a 5-lesson YouTube series by John Renfrew (FMTraining.tv), popular enough that Claris invited John to speak at Claris Engage 2025.
