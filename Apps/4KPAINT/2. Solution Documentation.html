Functional Documentation

This document describes every layout shown in the screenshots PDF, and it ties each screen to the feature area it belongs to.
It also adds light data-model context from the DDR XML so the whole solution makes sense as one system.

1. What the solution is for

This FileMaker solution runs day-to-day operations for a contractor business (painting, drywall, framing, etc).

It covers:

  • Setup of master data (Clients, Jobs, Employees, Contractors, Suppliers)
  • Daily job logging on site (supervisors and staff)
  • Approvals (hours approval, pay rate approval)
  • Reporting (Master Ledger, Consolidated Cost Reports)
  • Documents and contacts attached to the main records
  • A mobile-first workflow for iOS use on job sites

2. Core modules and how they connect

A. Dashboard and Navigation

Purpose:

  • Provide an at-a-glance operational view
  • Provide fast navigation into reporting and setup screens

Feeds into:

  • Ledger (cost and hours reporting)
  • Work Logs (daily logs and approvals)
  • Setup records (clients, jobs, employees, contractors, suppliers)

B. Ledger and Reporting

Purpose:

  • Report job cost, hours, and totals across time
  • Provide export options for sharing with management, accounting, or clients

Includes:

  • Master Ledger (primary report and drill style grouping)
  • Cost Reports / Consolidated Ledger (consolidated cost and hours view)

C. Work Logs and Payments

Purpose:

  • Capture what happened on site each day
  • Approve and control the data that becomes payroll and cost reporting

Includes:

  • Daily Log entry and review
  • Time Card approvals (hours approval)
  • Pay Rate approvals (hourly rate approval for payroll)

D. Setups

Purpose:

  • Maintain the reference records that drive logging and reporting
  • Keep contacts and documents tied to the correct parties

Includes:

  • Clients
  • Jobs
  • Employees
  • Contractors
  • Suppliers

E. iOS On-Site Workflow

Purpose:

  • Make daily logging usable on job sites
  • Keep entry fast and structured

Includes:

  • Daily Log Console (supervisor driven log sheet)
  • Daily Time Card (employee or contractor worker daily entry)

3. UI and UX standards to keep consistent

3.1 Uniform window sizes (your note, formalized)

Goal:

  • Every layout opens in a consistent window size and position.
  • Users do not feel like the system “jumps” in size from screen to screen.

Recommended standard:

  • Define a single “Desktop Standard” window size (example: 1440 x 900, or whatever matches the client’s common screen).
  • Define a single “iOS Standard” sizing rule for FileMaker Go layouts.

Implementation approach (conceptual):

  • On layout entry, run a standard “Adjust Window” routine.
  • Keep header height and footer height consistent.
  • Keep left list width consistent across all list-detail layouts.

3.2 Global navigation pattern

The app uses a hamburger menu pattern:

  • Users open the side menu from the top-left icon.
  • The menu provides direct access to the core modules.

This is important because the system is multi-module and users bounce between:

  • Daily logs
  • Approvals
  • Reporting
  • Setup records

3.3 Consistent tool sets

Across reporting and list screens, tools repeat:

  • Filter
  • Sort or grouping controls (where needed)
  • Export options: Excel, PDF, Print, Email
  • Record navigation: First, Previous, Next, Last (common on list screens)

This is good. The main improvement is to keep icon placement consistent per module.

9. Data model summary (high-level, based on DDR)

You included the DDR XML for the full solution. The important takeaway is the system is built around these core entities:

Main entities:

  • CLIENT and CLIENT_TYPE
  • JOB
  • EMPLOYEE and EMPLOYEE_TYPE
  • CONTRACTOR, CONTRACTORTYPE, and Contractor_Workers
  • SUPPLIER, SUPPLIERTYPE, SUPPLIERTERMS
  • DAILY_LOG
  • TIME_CARD
  • DOCUMENT and DOCUMENT_JOIN
  • COSTAREA and COSTCODE
  • USERS and USERS_SECURITY

Support tables exist for:

  • Globals (z_GLOBAL)
  • Help and key pairs (z_HelpFile, z_KeyPairs)
  • Virtual lists and UI helpers (zv_* tables)

Relationship picture in simple terms:

  • A Client has many Jobs
  • A Job has many Daily Logs
  • A Daily Log has many time entries (employees and contractor workers)
  • Time entries become Time Cards for approval
  • Approved time combines with approved pay rates for payroll calculations
  • Supplier invoices and contractor entries feed ledger reporting
  • Documents can attach to multiple record types through join tables

10. Your key product decisions, rewritten clearly

10.1 Ledger screens should be effortless to switch

You want Master Ledger and Consolidated Ledger to feel like two views of the same reporting area.

Recommended approach:

  • A single “Ledger” module screen
  • Two tabs inside it:
    • Master Ledger
    • Cost Reports (Consolidated Ledger)
  • Keep filter controls consistent where possible.
  • Keep export controls consistent and in the same place.

10.2 Work Logs and Payments should feel like one workflow

Daily Log is the entry layer.
Time Cards is the hours approval layer.
Pay Rates is the money approval layer.

That flow is already present. The goal is to keep navigation friction low:

  • A user reviewing a daily log should be able to jump directly to approvals.
  • A user in approvals should be able to jump back to the log details.

10.3 Setups should remain list-detail and predictable

Clients, Jobs, Employees, Contractors, Suppliers all follow the same mental model:

  • Select record on the left
  • Edit details on the right
  • Use tabs for supporting info (contacts, ledger, documents, security)

That consistency is a major strength. Protect it.

10.4 iOS screens should stay task-focused

On site, the user is trying to:

  • Pick the job
  • Add time fast
  • Add materials and notes when needed
  • Move to the next day or next job

The Daily_Log_Console screen does this well. The Time_Card_Daily_iPhone screen supports personal daily entry.

12. Closing note

This solution already has a clear structure:

  • Setup data feeds logs
  • Logs feed approvals
  • Approvals feed reporting
  • Mobile entry supports the real world workflow on site

If you keep those consistent patterns, the app will remain easy to learn and easy to run.
