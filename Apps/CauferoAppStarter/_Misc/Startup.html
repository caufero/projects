Startup Script Documentation
Canonical boot sequence for CauferoAppStarter. Runs at application launch to establish session globals, configure UI mode, load WebViewer shell, apply role-based navigation, and prepare the runtime environment. This script orchestrates the entire application initialization process.
Boot Sequence Session Init WebViewer Shell Role-Based Nav
📖 Overview Purpose and Responsibilities

Purpose

The Startup script is the canonical boot sequence for CauferoAppStarter that runs at application launch.

Core Responsibilities:

  • Setting the UI into expected "app mode" (kiosk-like look and feel)
  • Loading and caching global session variables for app, business, and user identity
  • Applying device and platform UI rules (zoom and WebViewer load strategy)
  • Preparing and rendering the WebViewer-based shell (Page and Menu)
  • Selecting initial navigation destination based on user role
  • Logging the user as "Logged In"
  • Resetting startup-only globals to leave system in clean state

Key Principle: This script does not "build the app". It configures the running session so every other script can rely on a stable set of global variables and a consistent UI environment.

Script Flow Summary

  1. Enter splash / pre-UI state
  2. Apply UI concealment (toolbars/menubar/ruler)
  3. Enable developer-friendly syntax coloring (MBS)
  4. Load core session data (App, Business, User)
  5. Set OAuth configuration for Google Drive
  6. Resolve theme and basic runtime defaults
  7. Apply privilege-based UI locking
  8. Maximize window
  9. Run splash animation (for non-Bridge apps)
  10. Log user as "Logged In"
  11. Render banner
  12. Load the shell layout and WebViewers
  13. Compute role-based first link and navigate
  14. Apply special post-load behavior for The Bridge
  15. Cleanup startup-only globals
🌐 Session Globals (Canonical) Runtime cache established by Startup

Startup establishes a set of $$ variables that act as the system's runtime cache. The system expects these globals to be available after Startup completes.

App Identity

  • $$App ID - Unique app identifier
  • $$App Name - Display name of the application
  • $$App Slogan - Marketing tagline
  • $$Partners - Partner organizations list
  • $$App Launch Year - Year app was launched

Splash Content

  • $$Splash Screen Loading Statement - Message shown during load
  • $$Splash Background Photo - Base64 or URL for splash image

Theme

  • $$App Default Theme - Default theme for the app
  • $$My Preferred Theme - User's theme preference
  • $$Theme - Resolved active theme

Business

  • $$Business Name - Company/organization name
  • $$Business Phone - Contact phone number
  • $$Business Location - Physical address
  • $$Website - Company website URL
  • $$Business Logo URL - Logo image URL
  • $$Business Logo Icon URL - Icon-sized logo URL
  • $$Menu Mode - "Persistent Side Menu" or other mode

User

  • $$My User ID - Current user identifier
  • $$My Staff ID - Linked staff record ID
  • $$My Staff Number - Employee number
  • $$My Fullname - User's full name
  • $$My First Name - User's first name
  • $$My Role ID - Role identifier
  • $$My Role - Role display name
  • $$My Photo - Base64 encoded profile photo

Preferences & UI

  • $$Preferred Currency - Default currency code
  • $$Work Mode - Current work mode state
  • $$Layout Name - Active layout context

Navigation & Utilities

  • $$Startup - Temporary flag during initial navigation (cleared after use)
  • $$No Image URL - Placeholder for missing images

OAuth (Google Drive Integration)

  • $$Google Client ID - OAuth client identifier
  • $$Google Client Secret - OAuth secret key
  • $$Google Scope - Requested OAuth permissions
🖼️ WebViewer Shell Model Layout and rendering strategy

CauferoAppStarter uses a WebViewer-based shell for the main UI.

Key WebViewer Objects

  • Page - The main content area
  • Links - The menu area (used when menu mode is persistent)

Shell Layout Selection

Startup ensures the correct shell layout is used:

  • Page With Menu layout when $$Menu Mode = "Persistent Side Menu"
  • Page layout otherwise

Platform-Specific WebViewer Loading

Different platforms require different approaches to loading HTML into WebViewers.

Startup uses:

  • data:text/html;base64, + Base64Encode ( $$Page ) and Base64Encode ( $$Menu ) when Get ( SystemPlatform ) = 4
  • Raw HTML strings otherwise

Note: The script treats Get ( SystemPlatform ) = 4 as the special case platform requiring base64 encoding.

🔍 Detailed Walkthrough Step-by-step breakdown

Step 1: Enter Startup UI Context (Splash Screen)

Startup begins by switching to the splash layout and hiding UI chrome:

  • Go to layout: Splash Screen (Settings)
  • Hide toolbars
  • Hide and configure menubar visibility/locking
  • Hide text ruler

Why: Ensures a clean launch state and prevents users from seeing FileMaker UI elements during initialization.

Step 2: Enable Syntax Coloring (MBS Plugin)

Startup calls MBS plugin functions to enable and configure syntax coloring for script steps, functions, and formula tokens.

Why: Improves readability of generated scripts or formulas shown in WebViewer-based text editors.

Step 3: Load App Details (Auxiliary Script)

Calls Perform Script [ "Get App Details" ]

This script populates:

  • $$App ID, $$App Name, $$App Slogan
  • $$Splash Screen Loading Statement
  • $$Partners, $$App Launch Year
  • $$Splash Background Photo
  • $$App Default Theme
  • Flushes caches via Refresh Window

Step 4: Apply Window Title

Set Window Title [ $$App Name ]

Why: Branding and usability cue when multiple FileMaker windows are open.

Step 5: Apply Zoom Based on Device

Calls Perform Script [ "Set Zoom Level Based On Device" ]

Applies device-specific zoom rules (75% for mobile, 100% for desktop with persistent menu, etc.)

Step 6: Load Current User Details

Calls Perform Script [ "Get My Details" ]

Populates user identity, role, theme preference, and profile photo globals.

Step 7: Set Google OAuth Globals

Sets OAuth configuration for Google Drive integration:

  • $$Google Client ID
  • $$Google Client Secret
  • $$Google Scope (Drive file scope, userinfo email/profile)

Step 8: Resolve Theme and Initialize Defaults

Sets $$Theme from user preference or app default, $$Work Mode = "Relaxing", and $$No Image URL placeholder.

Step 9: Privilege-Based UI Locking

If Get ( AccountPrivilegeSetName ) = "[Full Access]":

  • Menubar unlocked (but hidden)
  • Toolbars hidden

Else:

  • Menubar locked and hidden
  • Toolbars locked and hidden

Step 10: Maximize Window

Adjust Window [ Maximize ]

Why: Ensures consistent layout proportions.

Step 11: Splash Experience (Conditional)

If $$App Name ≠ "The Bridge":

  • Run +++ Splash Page
  • Pause 6 seconds

Step 12: Log User as Logged In

Calls Save A Record utility to set Logged In = 1 for current user.

Step 13: Set Initial Layout Name and Render Banner

Sets $$Layout Name = "Dashboard" then runs +++ Banner Page

Step 14: Load Shell Layout + WebViewers (Non-Platform-4)

If Get ( SystemPlatform ) ≠ 4:

  • If $$Menu Mode = "Persistent Side Menu": Go to Page With Menu, load both Page and Links WebViewers
  • Else: Go to Page

Step 15: Compute Role-Based First Link

Uses ExecuteSQL to query role links, ordered by Links::Order. Selects first permitted link.

Step 16: Trigger Initial Navigation

Sets $$Startup = 1, calls Select Link with first link ID, then clears $$Startup.

Step 17: Post-Load Layout Handling

If $$App Name = "The Bridge": Force re-render + refresh of Page and Links WebViewers.

Else if Get ( SystemPlatform ) = 4: Load shell using base64 strategy.

Step 18: Cleanup Startup Globals

Clears $$Splash, $$Splash Background Photo, $$Selected Record ID

⚙️ Auxiliary Scripts Supporting script responsibilities

Get App Details

Loads app identity and related configuration.

Operations:

  • Runs Preferred Currency
  • Runs Get Business Details
  • Uses ExecuteSQL to read from Apps by Settings::App ID
  • Assigns app globals (ID, Name, Slogan, Splash content, Theme, Partners, Launch Year)
  • Flushes caches

Preferred Currency

Loads the business's preferred currency into $$Preferred Currency

Source: Businesses::Preferred Currency filtered by Settings::Business ID

Get Business Details

Loads business identity and branding into globals.

Outputs:

  • $$Business Name
  • $$Business Phone
  • $$Business Location
  • $$Website
  • $$Business Logo URL
  • $$Business Logo Icon URL
  • $$Menu Mode (authoritative selector for shell layout)

Set Zoom Level Based On Device

Applies zoom rules based on device/platform/system conditions.

Examples:

  • If Get ( Device ) = 3: set zoom to 75%
  • Else if specific IP: set zoom to 75%
  • Else if $$Menu Mode = "Persistent Side Menu": set zoom to 100%

Get My Details

Loads current user identity, role, preferences, and profile image.

Outputs:

  • $$My User ID, $$My Staff ID, $$My Staff Number
  • $$My Fullname, $$My First Name
  • $$My Role ID, $$My Role
  • $$My Preferred Theme
  • $$My Photo (base64)

Flushes caches after loading.

📐 Agent Operating Rules How to interpret this documentation

Rule 1: Startup is Orchestration

Startup coordinates other scripts and configures the session. It should not contain heavy business logic.

Rule 2: Globals are the Contract

After Startup, downstream scripts should assume required globals exist. Do not duplicate the loading logic elsewhere.

Rule 3: Menu Mode Drives Shell Layout

  • If $$Menu Mode = "Persistent Side Menu": use Page With Menu and load both Page and Links
  • Otherwise: use Page layout

Rule 4: Role Determines First Page

The first page is computed by:

  • Role ID → Role Links → Links
  • Ordered by Links::Order
  • Must have Layout and Order set

Rule 5: Platform Differences are Deliberate

Maintain separate handling for:

  • Get ( SystemPlatform ) ≠ 4
  • Get ( SystemPlatform ) = 4 (uses base64 data: URLs for WebViewers)

Rule 6: Bridge is a Special Case

If app name equals "The Bridge", force re-render + refresh of Page and Links.

Rule 7: Clean State at End

Startup must clear transient variables so the system begins with a neutral context.
